## Running Tests

### Frontend Tests

<Tabs>
  <Tab title="Quick Run">
    ```bash
    # Run all tests once
    pnpm test
    ```
    
    This runs all frontend tests using Vitest.
  </Tab>
  
  <Tab title="Watch Mode">
    ```bash
    # Run tests in watch mode (re-runs on file changes)
    pnpm vitest
    ```
    
    Useful during development - tests automatically re-run when you save files.
  </Tab>
  
  <Tab title="Coverage Report">
    ```bash
    # Run tests with coverage
    pnpm vitest run --coverage
    ```
    
    Generates coverage reports in:
    - **Terminal:** Summary view
    - **HTML:** `coverage/index.html` (open in browser)
    - **LCOV:** `coverage/lcov.info` (for CI tools)
    - **JSON:** `coverage/coverage-summary.json`
  </Tab>
  
  <Tab title="Specific Tests">
    ```bash
    # Run tests matching a pattern
    pnpm vitest BoardGame
    
    # Run specific test file
    pnpm vitest src/features/boards/__tests__/BoardGame.test.tsx
    
    # Run tests in specific directory
    pnpm vitest src/features/profiles
    ```
  </Tab>
</Tabs>

### Backend Tests

<Tabs>
  <Tab title="All Tests">
    ```bash
    cd src-tauri
    
    # Run all tests
    cargo test
    ```
    
    Runs all 234 Rust tests (completes in ~3.4 seconds).
  </Tab>
  
  <Tab title="Specific Module">
    ```bash
    # Test specific module
    cargo test db::player_stats::tests
    
    # Test specific function
    cargo test test_position_search
    
    # Test with pattern matching
    cargo test puzzle
    ```
  </Tab>
  
  <Tab title="With Output">
    ```bash
    # Show println! output
    cargo test -- --nocapture
    
    # Show output for specific test
    cargo test test_name -- --nocapture
    ```
    
    Useful for debugging test failures.
  </Tab>
  
  <Tab title="Ignored Tests">
    ```bash
    # Run ignored tests (stress tests, benchmarks)
    cargo test -- --ignored
    
    # Run all tests including ignored
    cargo test -- --include-ignored
    ```
    
    <Note>
      Stress tests are ignored by default as they're performance benchmarks.
    </Note>
  </Tab>
</Tabs>

## Writing Frontend Tests

### Test Structure

Frontend tests use **Vitest** with **React Testing Library**.

<Steps>
  <Step title="Create Test File">
    Add tests alongside the component:
    ```
    features/boards/
    ├── components/
    │   └── BoardGame.tsx
    └── __tests__/
        └── BoardGame.test.tsx
    ```
  </Step>
  
  <Step title="Write Test">
    ```typescript
    import { render, screen, fireEvent } from '@testing-library/react';
    import { describe, it, expect, vi } from 'vitest';
    import { BoardGame } from '../components/BoardGame';
    
    describe('BoardGame', () => {
      it('renders chess board', () => {
        render(<BoardGame />);
        const board = screen.getByRole('main');
        expect(board).toBeInTheDocument();
      });
      
      it('handles move click', async () => {
        const onMove = vi.fn();
        render(<BoardGame onMove={onMove} />);
        
        const square = screen.getByTestId('square-e2');
        fireEvent.click(square);
        
        expect(onMove).toHaveBeenCalled();
      });
    });
    ```
  </Step>
  
  <Step title="Run Test">
    ```bash
    pnpm vitest BoardGame
    ```
  </Step>
</Steps>

## Writing Backend Tests

### Test Structure

Rust tests are typically in the same file as the code:

```rust
// src-tauri/src/db/search.rs

pub fn search_position(fen: &str) -> Result<Vec<Game>> {
    // Implementation
}

#[cfg(test)]
mod tests {
    use super::*;
    
    #[test]
    fn test_search_position() {
        let fen = "rnbqkbnr/pppppppp/8/8/8/8/PPPPPPPP/RNBQKBNR w KQkq - 0 1";
        let result = search_position(fen);
        assert!(result.is_ok());
    }
}
```

### Testing Patterns

<Tabs>
  <Tab title="Unit Tests">
    ```rust

    fn test_move_encoding() {
        let san_move = "e4";
        let encoded = encode_move(san_move).unwrap();
        let decoded = decode_move(encoded).unwrap();
        assert_eq!(decoded, san_move);
    }
    
    #[test]
    fn test_fen_parsing() {
        let fen = "rnbqkbnr/pppppppp/8/8/8/8/PPPPPPPP/RNBQKBNR w KQkq - 0 1";
        let position = parse_fen(fen).unwrap();
        assert_eq!(position.turn(), Color::White);
    }
    ```
  </Tab>
  
  <Tab title="Database Tests">
    ```rust

            black: "Player2".to_string(),
            moves: vec!["e4", "e5"],
    ```

## Continuous Integration

### GitHub Actions

Tests run automatically on:
- **Every commit** to any branch
- **Pull requests** before merge
- **Pre-release** builds

**Workflow:** `.github/workflows/test.yml`

```yaml
name: Test
on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Frontend tests
        run: pnpm test
      - name: Backend tests
        run: cd src-tauri && cargo test
```

## Best Practices

<CardGroup cols={2}>
  <Card title="Write Tests First" icon="pen-to-square">
    Consider TDD (Test-Driven Development):
    1. Write failing test
    2. Implement feature
    3. Test passes
  </Card>
  
  <Card title="Test Edge Cases" icon="triangle-exclamation">
    Don't just test happy paths:
    - Empty inputs
    - Invalid data
    - Boundary conditions
    - Error cases
  </Card>
  
  <Card title="Keep Tests Simple" icon="list-check">
    - One assertion per test (when possible)
    - Clear test names
    - Minimal setup
    - No complex logic in tests
  </Card>
  
  <Card title="Mock External Dependencies" icon="plug">
    - Mock Tauri commands
    - Mock external APIs
    - Use in-memory databases
    - Avoid network calls
  </Card>
</CardGroup>
